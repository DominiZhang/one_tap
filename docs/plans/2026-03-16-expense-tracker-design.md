# 记账 App（极速个人记账）设计文档

日期：2026-03-16  
主题：跨平台（Flutter / React Native）极速个人记账 App（MVP 设计）

## 1. 背景与目标

### 1.1 背景
多数记账失败的核心原因不是“功能不够强”，而是“记录成本太高”。本项目以“极速个人记账”为第一目标，通过默认值、联想与轻量信息架构，把一次记账压缩到 3 秒内完成。

### 1.2 成功标准（可量化）
- 新增一笔支出：从打开 App 到保存成功，目标 3 秒内完成（熟练用户）
- 80% 以上账目只填写“金额 + 类别”（账户/备注/时间保持可选且默认合理）
- 明细与概览在中等数据量（1-2 万条）下仍保持顺滑体验（后续优化点）

### 1.3 非目标（MVP 不做）
- OCR 小票识别
- 复杂财务报表体系（多维透视、资产负债表等）
- 完整云端账户体系与多端实时同步（可作为后续版本）

## 2. 用户画像与核心场景

### 2.1 用户画像
- 个人记账用户：希望“随手记、别麻烦”，不想被表单绑架
- 轻度分析需求：偶尔想看“本月花在哪儿了”，但不愿为此多做输入

### 2.2 核心场景（极速记一笔）
- 在餐后/通勤/购物当下快速记录
- 记录后偶尔回看明细、按类别大致检查支出结构

## 3. 方案对比与选择

### 3.1 方案 A：极简记一笔（推荐）
- 首页即“记一笔”主面板：金额输入为中心，类别选择极少点击
- 默认值与最近使用联想减少输入

### 3.2 方案 B：聊天式/自然语言记账
- “午饭 28 微信”自动解析
- 差异化强，但解析与纠错交互复杂，首版风险高

### 3.3 方案 C：拍照/票据识别
- 体验看似强，但准确率与成本、交互闭环复杂，首版不适合

**结论**：MVP 采用方案 A，先把“3 秒记账 + 自动联想 + 轻统计”做到极致；B/C 作为后续增强。

## 4. 信息架构与导航结构

### 4.1 Tab 导航（推荐 3 个）
- **记账**：默认入口（主面板）
- **明细**：时间倒序列表 + 快速编辑/复制
- **概览**：本月总览 + 类别排行 + 简单趋势

（可选后续）**我/设置**：类别/账户/导出/备份等

### 4.2 MVP 页面清单
- 记账页（支出/收入/转账切换）
- 明细页（列表、筛选/搜索可延后）
- 概览页（本月统计、Top 类别、趋势）
- 类别管理页
- 账户管理页
- 导出/导入（备份）页（可放设置里）

## 5. 关键交互设计（MVP）

### 5.1 记账页（主战场）
- **默认支出**；顶部切换：支出/收入/转账
- **金额输入为中心**：大号金额展示，数字键盘常驻
- **类别快捷网格**：
  - 默认显示“最近使用 + 常用”混排（减少寻找成本）
  - 提供“更多类别”入口进入完整类别选择
- **次要字段折叠**：账户/备注/时间位于可展开区域
- **一键保存**：金额有效且已选类别（或自动推荐类别）时，保存按钮高亮

### 5.2 明细页
- 日期分组：今天/昨天/本周/更早
- 列表项：类别图标 + 备注（无则空）+ 金额 + 账户标签
- 手势：
  - 左滑：编辑/删除
  - 长按：复制一笔（快速复用）

### 5.3 概览页（轻量）
- 本月支出总额、日均
- 类别 Top5（点击进入该类别明细/筛选结果）
- 7/30 天趋势（折线或柱状，保持轻量）

## 6. 数据模型（MVP）

### 6.1 Transaction（账目）
- `id`: UUID
- `type`: `expense | income | transfer`
- `amount`: 整数分（避免浮点误差）
- `currency`: 默认币种（MVP 可固定单币种）
- `categoryId`: 类别（支出/收入类别可分开）
- `accountId`: 账户（支出/收入）
- `fromAccountId`, `toAccountId`: 转账专用
- `note`: 备注（可空）
- `occurredAt`: 发生时间（默认 now）
- `createdAt`, `updatedAt`
- `deletedAt`（可选，软删除；利于撤销与未来同步）

### 6.2 Category（类别）
- `id`, `name`, `icon`, `color`
- `type`: `expense | income`
- `sortOrder`
- `isArchived`

### 6.3 Account（账户）
- `id`, `name`
- `type`: `cash | bank | wechat | alipay | custom`
- `sortOrder`
- `isArchived`

### 6.4 AppSettings
- 默认币种、周起始日、是否显示小数、快速类别布局等

## 7. 存储、备份与同步策略

### 7.1 本地存储（本地优先）
原则：所有操作先落本地，保证离线可用、秒开秒记。

**推荐**：SQLite（查询/聚合友好，生态成熟）  
备选：Realm/Isar（开发体验好，但需评估体积、生态与迁移策略）

### 7.2 查询与统计（MVP）
- 明细：按 `occurredAt` 倒序分页
- 概览：按月聚合 `SUM(amount)`，按类别分组（必要时可引入月度缓存表作为后续优化）

### 7.3 备份（MVP）
- **JSON 导出/导入**：包含 `schemaVersion`，支持未来迁移
- **CSV 导出**：方便用户用表格分析

### 7.4 云同步（后续版本）
可选路径：
- iCloud/Google Drive 文件同步：实现相对轻，但需做冲突处理
- 自建后端账户体系：能力最强，成本最高

基础冲突策略（如启用云）：
- LWW（last-write-wins）：以 `updatedAt` 为准
- 软删除优先：`deletedAt` 覆盖旧版本

## 8. 错误处理与可靠性
- 输入校验：金额必填且 > 0；转账需选择两账户且不同
- 数据一致性：写入账目与统计缓存（如有）应在同一事务内
- 迁移：数据库与导出数据都携带 `schemaVersion`

## 9. 测试策略（MVP）
- 单元测试：
  - 金额（分）处理与格式化
  - 账目增删改（含软删除逻辑如采用）
  - 月度聚合与类别分组正确性
- 集成测试：
  - DB migration
  - 导出→导入 round-trip 一致性

## 10. MVP 范围总结
MVP 必做：
- 支出/收入记账 + 明细列表 + 本月概览
- 类别/账户管理（增删改排序、归档）
- JSON/CSV 导出（备份）

MVP 延后：
- 搜索/复杂筛选（v1.1）
- 云同步（v1.5+）
- 自然语言/拍照识别（v2+）
